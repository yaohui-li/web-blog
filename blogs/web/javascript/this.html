# 令人头疼的 This 对象

在 Javascript 中 this 动态指向，场景较多。 开发中经常会遇见 this 指向谁的问题。理解 this 对我们开发并且写出更加优雅的代码是极有裨益的。

# This 到底指向谁呢?

# 1. 全局环境中的 this

在全局环境中调用时，this 指向 window; 在严格模式下(use strict)指向 undefined

function fn() {
  console.log(this); // window
}
function strictFn() {
  "use strict";
  console.log(this); // undefined
}

这个是一个基础的列子, 下面已改打印什么呢?

const foo = {
  bar: 100,
  fn: funciton () {
    console.log(this)
    console.log(this.bar)
  }
}
const fn = foo.fn
fn()
答案

window undefined

原因

fon.fn 函数赋值给 fn 变量, 在执行 fn 这个变量函数时,执行环境在全局环境中。因此 this 为 window 对象,而 window 对象下没有 bar 属性, 所以打印 undefined

如果换成另外一种调用的话, 结果会是怎样呢？

const foo = {
  bar: 100,
  fn: funciton () {
    console.log(this)
    console.log(this.bar)
  }
}
foo.fn()
答案

foo 100

# 2. 上下文对象调用中的 this

当上下文对象调用时，this 指向最后调用它的对象。看看下面的例子，你知道答案吗？

const o1 = {
  text: 1,
  print() {
    return this.text;
  },
};
const o2 = {
  text: 3,
  print() {
    return o1.print();
  },
};
const o3 = {
  text: 3,
  print() {
    const print = o1.print;
    return print();
  },
};
答案

1, 1, undefined

原因

第一个打印为 1，比较简单且在上面提到就不赘述了

第二个打印也为 1 return 的是 o1 调用 print 方法, 对于 print 方法里面的 this 而言, 将会指向 o1 对象

第三个打印为 undefined 将 o1 的方法 print 赋值于变量 print，由于脱离了 o1 的执行上下文裸奔调用, 此时的 this 指向全局对象 window

# 3. 通过 bind/call/apply 改变 this 指向

const foo = {
  name: "_ucas",
  logName: function () {
    console.log(this.name);
  },
};
const bar = {
  name: "mike",
};
console.log(foo.logName.call(bar)); // result -> mike

bind/call/apply 都是用来改变 this 指向的函数，它们之间有些区别.

区别

call/apply 直接执行相关函数调用, 第一个参数为 this 指向的对象,call/apply 之间的区别在于参数的设定上 bind 不会执行相关函数,则是返回一个新的函数并且设定了函数 this 的指向

# 4. 构造函数和 this

想了解构造函数的 this, 需要了解构造函数执行时,做了些什么？

  1. 创建一个新的对象
  2. 将构造函数的 this 指向这个新的对象
  3. 为新对象添加属性和方法
  4. 最终返回新的对象
function Foo() {
  this.bar = "Lucas";
}
const instance = new Foo();
console.log(instance.bar);

上述过程可以用如下代码表述:

const obj = {};
obj.__proto__ = Foo.prototype;
Foo.call(obj);

如果构造函数出现了显示 return 情况, 看看如下代码如何输出?

  • 场景一
function Sudent() {
  this.name = "Lucas";
  const o = {};
  return o;
}
const instance = new Foo();
console.log(instance.name);
  • 场景二
function Sudent() {
  this.name = "Lucas";
  return 1;
}
const instance = new Foo();
console.log(instance.name);
答案

undefined Lucas

分析

场景一: 由于构造函数实际也是函数,如果显示的 return 是一个对象的话 this 会指向 return 的那个对象

场景二: 显示 return 的是一个数字,即一个基本类型, this 仍会指向实例

# 5. 箭头函数中的 this

在箭头函数中, this 的指向由外层(函数/全局)作用域来决定的。

const foo = {
  fn: function () {
    setTimeout(function () {
      console.log(this);
    });
  },
};
console.log(foo.fn()); // result -> window

如果 setTimeout 的回调函数改为箭头函数呢？

const foo = {
  fn: function () {
    setTimeout(() => {
      console.log(this);
    });
  },
};
console.log(foo.fn()); // result -> fn函数

# 6. this 优先级

通常把 call、apply、bind、new 对 this 的绑定称为显式绑定，根据调用关系确定 this 的指向的情况称为隐式绑定

显式绑定和隐式绑定谁的优先级更高呢？看看一段代码示例

function foo() {
  console.log(this.a);
}

const o1 = {
  a: 1,
  foo: foo,
};

const o2 = {
  a: 2,
  foo: foo,
};
o1.foo.call(o2);
o2.foo.call(o1);
答案

2， 1

看了运行结果发现，call/apply 的显示绑定一般优先级更高。 在看一段代码

function foo(a) {
  this.a = a;
}

const o1 = {};

const bar = foo.bind(o1);

bar(2);

console.log(o1.a); // result -> 2

如果将 bar 作为构造函数调用时，输出结果又是另一个结果。

const baz = new bar(3);
console.log(baz.a); // result -> 3

原因

bar 函数本身通过 bind 方法将 this 的指向了 o1 对象,当再次作为构造函数通过 new 操作符执行时, 新实例和 o1 就已经解绑了, 也就是 new 操作符修改了 this 的指向, 由此可见 new 绑定的优先级高于 bind 绑定

再看看箭头函数, 如下代码示例:

var a = 123;
function foo() {
  return (a) => {
    console.log(this.a);
  };
}

const o1 = {
  a: 2,
};
const o2 = {
  a: 3,
};

var bar = foo.call(o1);
console.log(bar.call(o2)); // result -> 2

如果将 foo 函数改写成如下代码, 结果就会是 123。

const foo = () => {
  return (a) => {
    console.log(this.a);
  };
};

💡 如果变量 a 的 var 修改为 const 结果是什么❓

- var a = 123;
+ const a = 123;
答案

undefined

原因

const 声明的变量不会挂载到 window 对象上, 所以在 window 自然是找不到 a 变量的。
